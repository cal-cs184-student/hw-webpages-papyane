Part 1: Ray Generation and Scene Intersection

Ray Generation: We begin with a pixel coordinate in the image space then normalize it to be in the range [0,1]x[0,1]. Then within "generate_ray," we shift the points in order to place them in the center of the Field of View, before we scale them according to the field of view. Then using the camera to world matrix we move from camera to world vector, then normalize. This gives us the direction of our ray with the position being the camera's position. Once we can generate rays, we can sample camera rays around a pixel to estimate the scene radience around the pixel. This sampling and averaging around the pixel is used to anti-alias, smoothing out any staircase effects on the edges of triangles and spheres. Scene Intersection: To determine what a ray hits, we need to iterate through primitives like the triangle and sphere, checking if they intersection with the ray. To check the intersection, we must calculate the time at which the ray hits the surface, ensure t is within the rays range, if its valid then we set the max of the ray to the new t (making sure that future checks only look for objects which are closer than the current hit), and lastly we record the intersection by keeping track of the normal vector, the material, and the hit time using the Intersection struct. For triangle intersection in particular, we implemented Moller-Trumbore. This method essentially sets the ray equation (r(t) = origin + t * direction) equal to the barycentric equation of the triangle, which gives us t and u,v (the coordinates on the triangle's surface).

Part 2: Bounding Volume Hierarchy

The BVH construction algorithm takes in a start, end, and max leaf size, where start and end are iterators with Primitives. It begins by initializing a bounding box and adding all the primitives from the given start to end into the bounding box. This box is the parent which will encompass all the objects in this branch of the tree. We keep track of the number of primitives to then check if the number is less than or greater than the max leaf size. If it's less than the max leaf size, this bbox will be a leaf node. Otherwise, we have to recurse to create two child nodes where the number of primitives is split between the children. To do this, we find the splitting point along the longest axis, which helps with creating child bounding boxes as tight as possible. The heuristic we use is averaging the centroid of all the primitives. We collect the sum of the centroids of every axis of a primitive in the loop earlier where we populated the initialized bbox, therefore making it unnecessary to loop again. Then we pick out the centroid sum of the longest axis and divide by the number of primitives to get our splitting point. Then using the built in partition function we split the primitives from start to end based on if it's centroid is less than or greater than the splitting point. This is object partitioning! We then check the size of the primitives from start to middle and middle to end to ensure that if a side is empty, we force a split by the index. Finally we are ready to recurse! By calling the construct_bvh function from start to mid and mid to end, we get the left and right child respectively. We create our interior node, setting the children to what we got via recursion and returing that node.
