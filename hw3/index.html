Part 1: Ray Generation and Scene Intersection

Ray Generation: We begin with a pixel coordinate in the image space then normalize it to be in the range [0,1]x[0,1]. Then within "generate_ray," we shift the points in order to place them in the center of the Field of View, before we scale them according to the field of view. Then using the camera to world matrix we move from camera to world vector, then normalize. This gives us the direction of our ray with the position being the camera's position. Once we can generate rays, we can sample camera rays around a pixel to estimate the scene radience around the pixel. This sampling and averaging around the pixel is used to anti-alias, smoothing out any staircase effects on the edges of triangles and spheres.

Scene Intersection: To determine what a ray hits, we need to iterate through primitives like the triangle and sphere, checking if they intersection with the ray. To check the intersection, we must calculate the time at which the ray hits the surface, ensure t is within the rays range, if its valid then we set the max of the ray to the new t (making sure that future checks only look for objects which are closer than the current hit), and lastly we record the intersection by keeping track of the normal vector, the material, and the hit time using the Intersection struct. For triangle intersection in particular, we implemented Moller-Trumbore. This method essentially sets the ray equation (r(t) = origin + t * direction) equal to the barycentric equation of the triangle, which gives us t and u,v (the coordinates on the triangle's surface).

Part 2: Bounding Volume Hierarchy

The BVH construction algorithm takes in a start, end, and max leaf size, where start and end are iterators with Primitives. It begins by initializing a bounding box and adding all the primitives from the given start to end into the bounding box. This box is the parent which will encompass all the objects in this branch of the tree. We keep track of the number of primitives to then check if the number is less than or greater than the max leaf size. If it's less than the max leaf size, this bbox will be a leaf node. Otherwise, we have to recurse to create two child nodes where the number of primitives is split between the children. To do this, we find the splitting point along the longest axis, which helps with creating child bounding boxes as tight as possible. The heuristic we use is averaging the centroid of all the primitives. We collect the sum of the centroids of every axis of a primitive in the loop earlier where we populated the initialized bbox, therefore making it unnecessary to loop again. Then we pick out the centroid sum of the longest axis and divide by the number of primitives to get our splitting point. Then using the built in partition function we split the primitives from start to end based on if it's centroid is less than or greater than the splitting point. This is object partitioning! We then check the size of the primitives from start to middle and middle to end to ensure that if a side is empty, we force a split by the index. Finally we are ready to recurse! By calling the construct_bvh function from start to mid and mid to end, we get the left and right child respectively. We create our interior node, setting the children to what we got via recursion and returing that node.

Part 3: Direct Illumination

The uniform hemisphere sampling function takes in the current ray and hit intersection, then sets up a local coordinate space where the normal is aligned with the Z-axis. It initializes an output radiance vector and enters a loop that runs for a total number of samples determined by the number of lights in the scene multiplied by the area light sample count. In each iteration, we grab a random direction from the hemisphere sampler and transform it into world space to cast a shadow ray from our hit point, using a small epsilon to avoid self-intersection. If this ray hits an object, we retrieve the emission from that object's BSDF and multiply it by the current material's reflection and the cosine of the angle, then divide by the uniform PDF of 1/(2π). Finally, we return the average of all these samples to get our direct lighting estimate.

The importance sampling implementation similarly starts with a coordinate space setup but iterates through every individual light in the scene to sample them directly. For each light, we determine if it is a delta light to take either one sample or multiple area light samples, then call the light's sampling function to get the incoming radiance, world-space direction, distance, and a specific PDF. We convert the direction to object space to ensure the light is in front of the surface before casting a shadow ray that is capped at the distance to the light source. If the BVH confirms the path is not blocked, we calculate the contribution by weighting the radiance with the BSDF and the light's PDF. We accumulate these results for every light to produce a final radiance value that is significantly less noisy than the uniform method.

Part 4: Global Illumination

The indirect lighting function takes in a ray and an intersection, where the ray is the current bounce traveling through the scene and the intersection contains information about the surface it hit. It begins by setting up a local coordinate system at the hit point with the surface normal aligned to the Z axis, giving us the transformation matrices o2w and w2o to convert between object and world space. We also compute hit_p, the actual point in world space where the ray intersected the surface, and w_out, the outgoing direction toward the camera transformed into object space.

We then add the direct lighting contribution via one_bounce_radiance, but only when we are either accumulating all bounces (isAccumBounces = true) or we are at the deepest level (r.depth == 1). This condition handles both the normal global illumination case and the unaccumulated bounce renders needed for the write-up.

Next we check if there is any depth remaining. If r.depth <= 1 we have hit our recursion limit and return immediately, since there is nowhere left to bounce.

If we do have depth remaining, we sample a new incoming direction using the BSDF at the hit surface via sample_f. This gives us a new direction w_in in object space along with its probability density pdf and the BSDF value. If the pdf is zero or negative we bail out early to avoid a degenerate division.

This is where Russian Roulette comes in. We define a continuation probability cpdf = 0.65, meaning each bounce has a 65% chance of continuing. However, we always trace the very first indirect bounce unconditionally, which we track with the first_indirect flag by checking if we are still at max_ray_depth. For all subsequent bounces we flip a coin and terminate randomly if it fails. This gives us an unbiased estimator that can theoretically handle infinite bounces while still terminating in finite time.

If we survive Russian Roulette, we construct a new bounce ray from hit_p in the sampled world space direction, setting min_t = EPS_F to avoid self-intersection and decrementing the depth by one. We then check if this bounce ray intersects anything in the scene via the BVH. If it does, we recursively call at_least_one_bounce_radiance on the new intersection to get the indirect radiance arriving from that direction. Finally we apply the full rendering equation weight — multiplying by the BSDF value, the cosine of the angle (w_in.z since we are in object space), dividing by the pdf, and dividing by cpdf to correct for the Russian Roulette termination probability, keeping the estimator unbiased. This weighted indirect radiance is accumulated into L_out and returned.

Part 5: Adaptive Sampling

Adaptive sampling is a technique that avoids wasting samples on pixels that have already converged to a stable color. Instead of blindly shooting a fixed number of rays through every pixel, we concentrate samples in the harder parts of the image — like shadow boundaries and caustics — while stopping early on easy regions like flat walls or the ceiling.

The implementation lives entirely in raytrace_pixel. It takes in a pixel coordinate x and y and works by maintaining two running statistics as it traces samples: s1, the sum of all illuminance values seen so far, and s2, the sum of their squares. These two numbers are all we need to compute the mean and variance at any point without storing every individual sample.

For each sample we generate a random offset within the pixel using the grid sampler, scale it to the range [0,1] to get normalized image coordinates, and generate a camera ray. We set r.depth = max_ray_depth on the ray so the global illumination recursion has the correct depth budget. We then call est_radiance_global_illumination to get the radiance for that sample, accumulate it into our running sum, and convert it to a scalar illuminance via .illum() to update s1 and s2.

Rather than checking convergence after every single sample, which would be expensive, we check every samplesPerBatch samples. At each check we compute the mean as μ = s1/n and the variance as σ² = (s2 - s1²/n) / (n-1), which is the standard one-pass variance formula that falls directly out of our two running totals. From the variance we compute I = 1.96 * σ / sqrt(n), which represents the half-width of a 95% confidence interval around the mean. Intuitively, I shrinks as either the variance decreases or the sample count grows — both of which indicate the pixel is converging. If I <= maxTolerance * μ, we consider the pixel converged and break out of the loop early.

Finally we write the averaged color sum / samples_taken into the sample buffer and store the actual number of samples taken into sampleCountBuffer. That last part is important — by storing samples_taken rather than ns_aa, the renderer can generate the rate image showing red for pixels that needed many samples and blue for pixels that converged quickly, making it easy to see exactly where in the scene adaptive sampling is doing its work.
